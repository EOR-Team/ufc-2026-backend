# 索引系统管理原则

## 目的

`docs/index/` 是文档系统的**自引用索引目录**，其职责是为整个 `docs/` 树提供快速定位能力。索引系统由两个文件构成：

- `docs/index/RULE.md`（本文件）：定义索引系统的规则与约束
- `docs/index/index.md`：实际承载分类索引表的数据文件

**索引系统的存在意义**：当 Agent 进入项目时，通过阅读 `docs/index/index.md` 可以快速定位到所需规则文档，而无需遍历整个 `docs/` 树。

---

## 与 docs/RULE.md 的关系

本文件**继承自** `docs/RULE.md` 定义的通用文档元数据标准：

1. **frontmatter 继承**：索引系统文件必须遵循 `docs/RULE.md` 规定的 `name/category/field/description/date` 五字段标准
2. **category 定义继承**：本目录下所有文件 category 必须为 `infra-rule`（基础设施规则）
3. **field 定义继承**：本目录文件 field 必须为 `global`（文档系统级作用域）

**本文件在前述标准之上，增加索引系统专有的约束规则**，不替代 `docs/RULE.md` 的通用标准。

---

## 索引目录结构

```
docs/index/
├── RULE.md     # 本规则文件：定义索引系统的行为约束
└── index.md    # 索引数据文件：承载分类索引表
```

### 文件职责

| 文件 | 职责 | 性质 |
|------|------|------|
| `RULE.md` | 定义索引系统规则、触发条件、内容要求、反模式 | **规则（Rule）** |
| `index.md` | 实际存储分类索引表，映射类别到文件路径 | **数据（Data）** |

**关键约束**：`RULE.md` 与 `index.md` 必须严格分离：
- `RULE.md` 只写**规则**，不承载索引数据
- `index.md` 只写**索引表**，不重复规则内容

## 场景一：更新 index/index.md

当以下任一条件满足时，**必须**更新 `docs/index/index.md`：

### 更新触发器

1. **新增文档**：在 `docs/` 下新增任何 `.md` 文件
2. **删除文档**：从 `docs/` 下删除任何 `.md` 文件
3. **移动文档**：将 `.md` 文件迁移到不同子目录
4. **重命名文档**：修改 `.md` 文件名
5. **文档 category/field 变更**：修改文档的 frontmatter 元数据
6. **重要决策完成**：新增 ADR（`changelog/` 下新建条目）

### 更新流程

更新索引时，按以下步骤操作：

1. **检查触发条件**：确认本次操作是否触发索引更新（见上）
2. **定位目标位置**：根据文档 category 确定应插入到 `index.md` 的哪个分类章节
3. **填写索引条目**：按以下格式添加行
   ```
   | [文件名](./相对路径) | `category` | 用途/描述 |
   ```
4. **更新排序**：按文件名字典序排列同分类下的条目
5. **更新日期**：将 `index.md` 的 frontmatter `date` 改为当天日期
6. **验证完整性**：确认索引表覆盖所有 `docs/` 下的 `.md` 文件（排除本目录）

### 索引条目格式规范

| 字段 | 要求 | 示例 |
|------|------|------|
| 文件名 | 使用 `[标题](./路径)` 格式，标题与 frontmatter `name` 保持一致 | `[RULE.md](./RULE.md)` |
| category | 使用**反引号**包裹，值必须与文档 frontmatter 一致 | `` `infra-rule` `` |
| 描述 | 一句话概括文档核心用途，简洁明了 | `定义 frontmatter Schema 与修改要求` |

**错误示例**：
```
| RULE.md | infra-rule | 规则文档  # 错误：未使用链接格式
| [file](./file.md) | infra-rule | 这是描述而且很长很长很长很长很长很长很长很长很长很长  # 错误：描述超过一行
```

## 场景二：修改 index/RULE.md 本身

当需要修改本规则的任何条款时：

1. **判断影响范围**：修改是否影响已有索引条目的有效性？
2. **更新日期**：将 frontmatter `date` 改为修改当天
3. **同步通知**：如果修改涉及索引结构变化，需要同步更新 `index/index.md` 的分类组织方式

---

## 索引内容要求

### 有效索引的判断标准

一个有效的索引条目**必须**满足：

1. **路径可解析**：链接的相对路径必须指向真实存在的文件
2. **category 匹配**：索引表中记录的 category 必须与被索引文件 frontmatter 一致
3. **非空描述**：每个条目必须有描述内容，禁止留空
4. **无重复条目**：同一文件不得在索引表中出现多次

### 索引分类组织原则

`index.md` 按以下层次组织分类：

1. **一级分类**：按 category 划分大章节（如 `General Rule`、`Specific Rule`、`ADR`）
2. **二级分类**：同 category 下按功能/用途进一步细分
3. **快速参考**：末尾提供按需求查询的交叉引用表

### 必含分类

`index.md` **必须**包含以下分类章节：

| 分类 | 说明 |
|------|------|
| General Rule | 文档基础设施规则（`docs/RULE.md`、`docs/memory/RULE.md` 等） |
| Specific Rule | 项目开发规范（安全、代码风格、TDD 等） |
| ADR | 架构决策记录（`changelog/` 下的所有条目） |
| Archive | 归档文档（历史研究、故障排查等） |
| 快速参考 | 按需求查询的交叉引用表 |

## 反模式

以下行为**禁止**出现在索引系统中：

### 禁止行为

1. **禁止在 RULE.md 中存放索引数据**
   - 规则文件只写规则，不承载索引表内容
   - 混淆会导致规则难以维护和定位

2. **禁止在 index.md 中写规则条款**
   - 索引文件只写分类路径映射，不写行为约束
   - 在索引表中夹杂规则会导致查找效率降低

3. **禁止索引条目指向不存在的文件**
   - 每次更新必须验证路径有效性
   - 死链接会误导 Agent，降低索引可信度

4. **禁止 category/field 与被索引文档不一致**
   - 索引表必须忠实反映被索引文档的 frontmatter
   - 不一致会导致分类失效

5. **禁止删除归档文档的索引条目**
   - 已归档文档的索引应保留，标记为 Archive 分类
   - 索引的历史追溯价值在于保留完整路径

6. **禁止在 index.md 中使用绝对路径**
   - 必须使用相对于 `docs/index/` 的相对路径
   - 绝对路径不可移植，破坏索引的跨环境一致性

7. **禁止索引覆盖不完整**
   - `docs/` 下所有 `.md` 文件（除 `index/` 目录自身）都应在索引中有对应条目
   - 遗漏会导致某些文档无法被快速定位

## 与其他 RULE 文件的交叉引用

| 被引用文件 | 引用关系 |
|------------|----------|
| `docs/RULE.md` | 定义 frontmatter 标准，本文件继承其规范 |
| `docs/memory/RULE.md` | 同为 `infra-rule`，但作用域限于 Memory Bank |
| `docs/changelog/RULE.md` | 定义 ADR 条目规范，ADR 文档入索引时需遵循 |
| `docs/rule/SECURITY.md` | 安全规则文档，索引中归于 `policy` category |
| `docs/rule/TDD.md` | TDD 工作流指南，索引中归于 `guide` category |

## 字段定义

| 字段 | 必填 | 说明 |
|------|------|------|
| `name` | 是 | `索引系统管理原则`（固定值） |
| `category` | 是 | `infra-rule`（固定值，继承自 docs/RULE.md） |
| `field` | 是 | `global`（固定值，适用于整个文档系统） |
| `description` | 是 | 一句话概括本规则约束对象 |
| `date` | 是 | 最后更新日期，修改内容后必须同步 |
