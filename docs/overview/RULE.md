# 项目概览管理原则

## 目的

`docs/overview/` 是 AI 世界观的**基线建立目录**，其职责是为 Agent 在项目任意操作前建立正确的世界观。概览系统由两个文件构成：

- `docs/overview/RULE.md`（本文件）：定义概览内容的维护规则与约束
- `docs/overview/project_overview.md`：实际承载项目世界观基线的数据文件

**核心原则**：如果世界观不正确，后续所有构建都会偏离目标。

---

## 与 docs/RULE.md 的关系

本文件**继承自** `docs/RULE.md` 定义的通用文档元数据标准：

1. **frontmatter 继承**：概览系统文件必须遵循 `docs/RULE.md` 规定的 `name/category/field/description/date` 五字段标准
2. **category 定义继承**：本目录下 RULE 文件 category 必须为 `infra-rule`（基础设施规则）
3. **field 定义继承**：本目录文件 field 必须为 `global`（文档系统级作用域）

**本文件在前述标准之上，增加概览系统专有的约束规则**，不替代 `docs/RULE.md` 的通用标准。

---

## 概览目录结构

```
docs/overview/
├── RULE.md              # 本规则文件：定义概览内容的维护约束
└── project_overview.md  # 概览数据文件：承载项目世界观基线
```

### 文件职责

| 文件 | 职责 | 性质 |
|------|------|------|
| `RULE.md` | 定义概览维护规则、触发条件、准确性要求、反模式 | **规则（Rule）** |
| `project_overview.md` | 实际存储项目世界观基线，包含能力、边界、技术栈、入口点 | **数据（Data）** |

**关键约束**：`RULE.md` 与 `project_overview.md` 必须严格分离：
- `RULE.md` 只写**规则**，不承载概览数据
- `project_overview.md` 只写**世界观内容**，不重复规则条款

---

## 场景一：更新 project_overview.md

当以下任一条件满足时，**必须**更新 `docs/overview/project_overview.md`：

### 更新触发器

1. **核心能力变更**：项目新增或删除功能模块
2. **技术栈更新**：依赖库版本变更、新增或移除依赖
3. **核心入口点变化**：`src/` 下新增模块、模块重命名、模块删除
4. **项目边界调整**：范围范围变化、外部依赖变更

### 更新流程

更新概览时，按以下步骤操作：

1. **检查触发条件**：确认本次变更是否影响概览内容的准确性
2. **定位变更点**：确定是能力描述、技术栈、入口点还是边界定义需要更新
3. **更新对应内容**：修改 `project_overview.md` 中受影响的部分
4. **同步验证**：
   - 技术版本号必须与 `requirements.txt` 一致
   - 路径描述必须指向真实存在的文件或目录
5. **更新日期**：将 `project_overview.md` 的 frontmatter `date` 改为当天日期
6. **同步通知**：如果概览变更影响索引中的描述，需同步更新 `docs/index/index.md`

### 更新原则

- **准确性优先**：概览描述必须与实际代码行为一致，禁止添加未经核实的描述
- **保守扩展**：只记录已实现的功能，不记录计划中或讨论中的功能
- **标记废弃**：已废弃的能力应标记为 `已废弃`，而非删除，以保持历史追溯性

---

## 场景二：修改 overview/RULE.md 本身

当需要修改本规则的任何条款时：

1. **判断影响范围**：修改是否影响概览数据的有效性？
2. **更新日期**：将 frontmatter `date` 改为修改当天
3. **同步通知**：如果修改涉及概览结构变化，需要同步更新 `project_overview.md` 的组织方式

---

## 内容准确性要求

### 有效概览的判断标准

一份有效的 `project_overview.md` **必须**满足以下准确性标准：

1. **技术版本一致**：文档中引用的依赖版本号必须与 `requirements.txt` 一致
2. **路径可解析**：文档中描述的文件路径必须指向真实存在的文件或目录
3. **能力描述匹配**：能力描述必须与实际 API 行为一致，禁止描述未实现的功能
4. **边界描述准确**：范围外事项的描述必须准确反映项目的实际边界
5. **入口点有效**：核心入口点必须是可以实际访问的模块或文件

### 概览内容组织原则

`project_overview.md` 按以下层次组织内容：

1. **能力定义**（项目做什么）：使用表格列出核心能力
2. **边界定义**（项目不做什么）：明确范围外事项
3. **技术实现**（用什么做）：技术栈表格
4. **入口点说明**（从哪里做）：模块职责表格
5. **状态评估**（当前稳定性）：已稳定 / 发展中 / 实验性

### 必含内容

`project_overview.md` **必须**包含以下章节：

| 章节 | 说明 |
|------|------|
| 项目做什么 | 核心能力表格 |
| 项目边界 | 范围内/范围外/外部依赖 |
| 技术栈 | 层级 + 技术 + 用途 |
| 核心入口点 | 模块 + 职责 |
| 状态 | 当前稳定性评估 |

## 反模式

以下行为**禁止**出现在概览系统中：

### 禁止行为

1. **禁止在 RULE.md 中存放概览数据**
   - 规则文件只写规则，不承载世界观内容
   - 混淆会导致规则难以维护和定位

2. **禁止在 project_overview.md 中写规则条款**
   - 概览文件只写世界观基线，不写行为约束
   - 在概览中夹杂规则会导致概念混淆

3. **禁止概览描述与代码实际行为不一致**
   - 每次更新必须验证描述与代码的一致性
   - 世界观错误会导致 AI 生成偏离目标的解决方案

4. **禁止描述范围外能力**
   - 只记录已实现的功能，不记录计划中或讨论中的功能
   - 虚假能力描述会误导 AI 决策

5. **禁止删除已废弃能力的历史记录**
   - 已废弃的能力应标记为 `已废弃`，而非删除
   - 历史追溯价值在于保留能力演化的上下文

| `docs/index/RULE.md` | 同为 `infra-rule`，但作用域限于索引系统 |
| `docs/index/index.md` | 概览在索引中归于 `concept` category |

## 字段定义

| 字段 | 必填 | 说明 |
|------|------|------|
| `name` | 是 | `项目概览管理原则`（固定值） |
| `category` | 是 | `infra-rule`（固定值，继承自 docs/RULE.md） |
| `field` | 是 | `global`（固定值，适用于整个文档系统） |
| `description` | 是 | 一句话概括本规则约束对象 |
| `date` | 是 | 最后更新日期，修改内容后必须同步 |
